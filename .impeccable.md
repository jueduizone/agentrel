# AgentRel Design Context

## Design Context

### Users
AI engineers and Web3 developers who need to give their agents reliable, up-to-date Web3 context. They access skills programmatically via API or browse to discover coverage. They spend time in terminals, dark editors, and documentation — they are sensitive to craft quality and immediately distrust anything that looks like a template or a SaaS marketing site. They evaluate infrastructure by whether it feels like it was built by people who ship.

### Brand Personality
**Web3 · AI · Open Source**

Translated into design language: **precise, composable, underground.**

Not corporate. Not playful. Not startup-marketingy. The interface should feel like a tool that belongs on a senior engineer's second monitor — dense enough to be useful, refined enough to signal it was built with intention.

Emotional goal: within 5 seconds, a visitor thinks "this is serious, I need this."

### Aesthetic Direction
**Linear × Vercel × Privy** — hardcore developer tool with visual ambition.

- From **Linear**: pixel-perfect detail, restrained palette that never feels safe, information density as a virtue not a problem to solve, subtle surface differentiation without gradients-as-decoration.
- From **Vercel**: black/white minimal with genuine code aesthetic, dual-theme that feels equally considered in both modes, typography that makes monospace beautiful rather than just utilitarian.
- From **Privy**: Web3-native dark atmosphere, light effects used with purpose (not as filler), the sense that the product exists at the edge of something new.

**Theme:** Both light and dark, each a first-class citizen — not light with a dark toggle as an afterthought. Dark mode should feel native and intentional; light mode crisp and precise, not "default browser."

**Anti-references:** Salesforce (too corporate, too much blue), Wix (template feel, generic cards, soft gradients).

**Typography approach:** Pair a distinctive display/heading face (not Inter, not the reflex list) with a tight, technical body face. Monospace elements (skill IDs, API endpoints, ecosystem tags) are design elements — lean into them rather than hiding them. The font must feel like it was *chosen*, not defaulted.

**Color:** Deep neutrals tinted subtly toward brand hue — not AI cyan, not generic purple. Accents should be rare and sharp. Both themes share the same chromatic personality, just inverted in weight.

**Code as chrome:** Skill IDs like `solana/web3js-v2`, version numbers, ecosystem tags, API endpoint strings — these are the visual texture of the interface. Don't sanitize them into soft badges. Let them look like what they are.

### Design Principles

1. **Infrastructure confidence.** Every detail — spacing, type weight, border thickness — signals that this was built by engineers who care about craft. Nothing soft, nothing generic, nothing that could appear in a Notion template.

2. **Density is a feature.** Developer tools don't hide data. Show the IDs, the version numbers, the schema. Information density, well-organized, is what makes a tool feel powerful rather than a toy.

3. **Dual theme as identity.** Both light and dark modes are equally considered. Switching themes should feel like flipping a different facet of the same precise object — not like a color-inversion hack.

4. **Restraint with edges.** Minimal decoration, but what's there is sharp. One strong typographic choice beats ten visual flourishes. If something doesn't earn its place, remove it.

5. **Code is aesthetic.** Monospace strings, API paths, ecosystem IDs — these are the "chrome" of the product. Treat them as typographic elements, not technical debt to be softened.
